# Parallel Dataset Generation

**Date:** 2026-06-05
**Status:** Approved

---

## Context

The evaluation dataset is generated by `evaluation/generate_dataset.ts`, which calls `runner/index.ts`. Today the runner executes each configured run sequentially: conversation 1 finishes before conversation 2 starts. With 7 planned runs at 4 turns each, all 28 LLM calls are serialised end-to-end.

Each run is fully independent — it has its own characters, scenario, alias map, and output file. There is no shared state between runs. The sequential execution is an implementation accident, not a requirement.

This spec replaces the sequential loop with full parallelism: all runs start simultaneously, each managing its own turn sequence internally.

---

## Goal

Reduce dataset generation wall time from `sum(run durations)` to `max(run duration)` with no change to output format, config schema, or any other part of the evaluation pipeline.

---

## What Does Not Change

- `conversation.ts` — run logic, turn loop, retry behaviour: untouched
- `config.ts` — no new fields; `concurrency` is not added (parallelism is unconditional)
- `writer.ts` — file format and naming: untouched
- All downstream consumers (judge, reconstruct): untouched
- Output directory structure and file contents: identical

---

## Execution Model

Replace the `for...of` loop in `index.ts` with `Promise.all`:

```typescript
await Promise.all(config.runs.map(run => executeRun(run, ...)))
```

All runs start simultaneously. Within each run, turns remain sequential — turn N depends on the messages from turns 1..N-1, so they cannot be parallelised. Across runs, full concurrency is safe because runs share no state.

No concurrency limit is applied. With the current dataset (7 runs), simultaneous requests to OpenRouter are well within provider limits.

---

## Failure Behaviour

Identical to today: **fail-fast, all-or-nothing**.

- Each run has 3 retry attempts (existing logic, unchanged)
- If any run exhausts its retries, `Promise.all` rejects immediately
- The outer `try/catch` catches the rejection, calls `rmSync` on the output directory, and re-throws
- The process exits with a non-zero code and no partial results are kept

This matches the current semantics: a broken dataset is useless, so there is no value in keeping partial output.

---

## Progress Output

Concurrent runs cannot print `[1/7]…✓` in order. Each log line is prefixed with the run label so interleaved output is readable:

```
[1/7] scenario_020 · char_001+char_007 — started
[2/7] scenario_016 · char_009+char_010 — started
[3/7] scenario_026 · char_001+char_003 — started
...
[3/7] scenario_026 · char_001+char_003 ✓
[1/7] scenario_020 · char_001+char_007 ✓
[2/7] scenario_016 ✗ attempt 1/3 — retrying
[2/7] scenario_016 ✓
...
Completed.
```

When a run fails after all retries, the label and error are logged before the rejection propagates to `Promise.all`.

---

## Architecture

Only `runner/index.ts` changes. The `for...of` loop (~25 lines) becomes a `Promise.all` call. An `executeRun` helper is extracted to hold the per-run retry logic and logging — this already exists implicitly inside the loop body; extraction makes the parallel structure legible.

```
evaluation/
  runner/
    index.ts      ← only file that changes
    conversation.ts   (unchanged)
    config.ts         (unchanged)
    writer.ts         (unchanged)
```

---

## Tests

No new test file. The existing `runner/__tests__/` covers config validation and conversation logic. A note is added to `config.test.ts` confirming that `concurrency` is intentionally absent from the schema.
